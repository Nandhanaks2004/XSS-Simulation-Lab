What is JavaScript?
Understanding the language that powers the web
JavaScript runs inside your browser and controls everything dynamic on a webpage. It can read, modify, and execute actions — all without user permission.
WHAT IT CAN DO
  • Show popup alerts
  • Validate forms
  • Update page content
  • Read cookies & storage
  • Make network requests
  • Control the entire page
WHERE IT RUNS
  • Inside your browser
  • Not on the server
  • Same as the website
  • Access to DOM & cookies
  • Can read your session
WHY IT'S POWERFUL
  • Trusted by the browser
  • No additional prompts
  • Runs silently
  • Full page access
TRY IT — CLICK THE BUTTONS BELOW

These buttons trigger real JavaScript. This is exactly how scripts work on any webpage.

console
KEY TAKEAWAY

JavaScript is not sandboxed from the page it runs on. A script on a page can access everything that page has — cookies, form data, local storage, and more.

How Browsers Trust JavaScript
The security model that XSS exploits
The browser's rule is simple: "If a script is inside the page → execute it." The browser doesn't question where the script came from or what it does.
THE ANALOGY

Think of a building with a security guard at the front door.

🏢 Building = the website
💂 Guard = the browser
🧑 Person inside = any script on the page
The guard trusts anyone already inside — no matter how they got there.
THE TRUST RULE
Script is part of the page HTML
↓
Browser receives the full page
↓
Browser sees <script> tag
↓
Executes script — no questions asked
WHAT THE BROWSER SEES
page source
<!-- Normal HTML (browser renders this) -->
<p>Welcome, Alice!</p>
<!-- Script inside the page (browser EXECUTES this) -->
<script>
  document.cookie // browser trusts this, gives full access
</script>
THE PROBLEM

The browser cannot tell the difference between:

✓ LEGITIMATE
A script the developer wrote intentionally
✗ INJECTED
A script an attacker injected via user input

What is XSS?
Cross-Site Scripting — the attacker's backdoor
XSS (Cross-Site Scripting) happens when a website allows user input to become JavaScript code that executes in the browser.
THE CORE LOGIC
User types input
→
Website displays it
→
Browser executes it
→
Attack succeeds
NORMAL BEHAVIOR
User types text → website shows text
Input: "Hello World"
Stored: "Hello World"
Displayed: Hello World
✓ Safe — text is shown as text
XSS ATTACK
User types script → browser runs it
Input: <script>alert(1)</script>
Stored: <script>alert(1)</script>
Executed: ⚠ popup fires!
✗ Dangerous — code ran as code
REAL-WORLD CONTEXT
SEARCH BOXES
Search results pages reflect your input back. A vulnerable site will execute injected scripts.
COMMENT SECTIONS
Comments get stored and shown to all users. Malicious scripts attack every visitor.
PROFILE FIELDS
Username, bio, location fields — anything that gets displayed can potentially carry XSS.

Types of XSS
Three distinct attack vectors
Reflected XSS
Stored XSS
DOM-based XSS
REFLECTED XSS — "ONE-TIME MIRROR"

The malicious input comes from the URL, gets processed by the server, and is reflected back in the response. Not stored — victim must click the crafted URL.

Attacker crafts malicious URL
↓
Victim clicks the link
↓
Server reflects input in response
↓
Browser executes script

The malicious script is saved to the database and executed for every user who views the page. One attack affects all visitors.

DOM-BASED XSS — "CLIENT-SIDE INJECTION"

The attack never touches the server. JavaScript in the page reads from the URL and writes it directly into the DOM. No server-side interaction needed.

SCRIPT TAG
<script>alert(1)</script>
The most basic XSS payload. The browser sees the <script> tag and immediately executes the JavaScript inside it. This works when the input is reflected inside HTML body content.
HOW IT WORKS
1. Attacker injects <script> into input
2. Server stores/reflects without encoding
3. Browser parses HTML, finds script tag
4. JavaScript executes in page context

Look for search boxes, comment fields, URL parameters, profile fields — anywhere input is displayed back.
💉
"Can I inject a script tag?"
Test with <script>alert(1)</script>. If a popup fires, the site is vulnerable.

"What data can I steal?"
document.cookie for session tokens. localStorage for auth data. keystrokes from forms.

Prevention
Stopping XSS at every layer
Golden Rule: "User input must never be executed as code." Treat all user input as untrusted data — encode it before displaying.
❌ VULNERABLE CODE
JavaScript — UNSAFE
// DANGEROUS: directly writing user input to DOM
element.innerHTML = userInput; // ← XSS possible!
// Input: <script>alert('hacked')</script>
// Result: script EXECUTES
✓ SECURE CODE
JavaScript — SAFE
// SAFE: textContent never executes HTML
element.textContent = userInput; // ← XSS blocked
// Input: <script>alert('hacked')</script>
// Result: shown as literal text ✓
LIVE COMPARISON: innerHTML vs textContent

OUTPUT ENCODING

Convert special characters to HTML entities. < becomes &lt; — browser shows it as text, never executes it.

textContent, encodeHTML()
INPUT VALIDATION

Reject input that doesn't match expected format. A phone number field should only accept digits — reject anything else.

allowlist validation
CONTENT SECURITY POLICY

HTTP header that tells the browser which scripts are allowed to run. Injected scripts from unknown sources get blocked.

Content-Security-Policy header
HTTPOnly COOKIES

Setting cookies as HTTPOnly prevents JavaScript from reading them. Even if XSS fires, it cannot steal the session cookie.

Set-Cookie: session=x; HttpOnly
FRAMEWORKS & LIBRARIES

React, Vue, Angular all auto-escape output by default. They treat all user data as text, never as HTML — XSS-safe by default.

React JSX, Angular templates
AVOID DANGEROUS APIs

Never use innerHTML, document.write(), or eval() with user data. These directly execute whatever you give them.

use textContent instead
